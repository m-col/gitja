    1  Documentation
    2  =============
    3  
    4  `gitserve` is a tool that reads from locally hosted or cloned git repositories
    5  and generates static HTML pages using their data.
    6  
    7  Usage
    8  -----
    9  
   10  ### Configuration file
   11  
   12  gitserve is configured using a [dhall](https://dhall-lang.org) config file. The
   13  [example config](https://github.com/m-col/gitserve/blob/master/config.dhall) is
   14  a good place to start, or alternatively create one in the current folder by
   15  running `gitserve -t`.
   16  
   17  It requires the following settings:
   18  
   19  | Setting    | Description                                                     |
   20  | ---------- | --------------------------------------------------------------- |
   21  | `repos`    | A list of folders containing your git repositories.             |
   22  | `scan`     | Whether `repos` lists repos or folders containing nested repos. |
   23  | `template` | The folder containing the template (see below).                 |
   24  | `output`   | Where to put the generated files.                               |
   25  | `host`     | The host URL, which is needed for creating links.               |
   26  
   27  If `scan` is `True`, then gitserve will look for git repositories in folders
   28  nested within those listed in `repos`. Otherwise, the folders in `repos` are
   29  assumed to be repositories themselves.
   30  
   31  Then, pass the config file to gitserve.
   32  
   33  ### CLI
   34  
   35  See the CLI help message for usage:
   36  
   37      Usage: gitserve [-c|--config CONFIG] [-q|--quiet] [-f|--force] [-t|--template] 
   38                      [-v|--version]
   39        üêô Templated web page generator for your git repositories
   40  
   41      Available options:
   42        -c,--config CONFIG       Configuration file to use (Default: ./config.dhall).
   43        -q,--quiet               Suppress non-error output.
   44        -f,--force               Force regeneration of all files.
   45        -t,--template            Create a template and config in the current folder.
   46        -v,--version             Print the gitserve's version.
   47        -h,--help                Show this help text
   48  
   49  Note the `force` flag. By default, gitserve will not generate new output for
   50  commits to save time. This flag will force regeneration of all files, which
   51  would be needed if changes have been made to the template.
   52  
   53  Templates
   54  ---------
   55  
   56  A base template can be created in the current folder by running `gitserve -t`.
   57  This can be used as a starting point to creating your own template.
   58  
   59  ### Folder structure
   60  
   61  The files that gitserve outputs are generated using
   62  [ginger](https://ginger.tobiasdammers.nl) template files, which use the
   63  [Jinja](https://jinja.palletsprojects.com) templating language. If these are
   64  new to you, it may be enough to skim through some of the examples in the
   65  [templates](https://github.com/m-col/gitserve/tree/master/templates) folder,
   66  otherwise the ginger docs can be very helpful to see what is supported.
   67  
   68  Templates are a folder containing a number of ginger template files.  There are
   69  4 "scopes", each making available a unique set of variables storing information
   70  about the git repositories. Each template file has access to a single one of
   71  these scopes. The structure of the template folder determines the scopes of the
   72  files contained therein.
   73  
   74  To illustrate, this is the expected structure:
   75  
   76      template/
   77          i_can_have_any_name.html
   78          and_there_can_be_any_number.html
   79          some_might_be_ginger_includes.html.include
   80          non_html_is_fine.css
   81          repo/
   82              inside_this_folder.html
   83              two_names_are_special.html
   84              file.html
   85              commit.html
   86  
   87  The top-level folder, here `template`, is that which is specified in the config
   88  file.
   89  
   90  Files ending in ".html" directly within that folder have access to the *index
   91  scope*, and are each parsed exactly once and output into the output directory
   92  with the same name.
   93  "[include](https://ginger.tobiasdammers.nl/guide/syntax/statements/#include)"
   94  files are never copied but can be used to assist in templating.
   95  
   96  The special folder "repo" has access to the *repo scope*, which exposes
   97  information pertaining to a single git repository. The template files contained
   98  within this folder are parsed and output once per git repository.
   99  
  100  The exceptions to this are the two special template files with the names
  101  "file.html" and "commit.html". These have access to the *file scope* and
  102  *commit scope* respectively, and are parsed and output once per file or commit.
  103  
  104  The resulting folder structure found in `output` will look like this (if
  105  `repos` only contains gitserve):
  106  
  107      output/
  108          i_can_have_any_name.html
  109          and_there_can_be_any_number.html
  110          non_html_is_fine.css
  111          gitserve/
  112              inside_this_folder.html
  113              two_names_are_special.html
  114              file/
  115                  LICENSE.html
  116                  Makefile.html
  117                  ...
  118              commit/
  119                  0a18f38bb5c398bd192a6268281fc6abefaedd63.html
  120                  0a7601059956d9c4d395f5d08e8cf48a515d080f.html
  121                  ...
  122          ...
  123  
  124  ### Static files
  125  
  126  Files in the template folder that do not end in ".html" or ".include" or are
  127  symbolic links, as well as directories that are not called "repo", are
  128  considered static content. These are copied unchanged (i.e. symbolic links are
  129  copied as symbolic links) to the output folder. This is useful for putting CSS,
  130  images etc into the output.
  131  
  132  Similarly, any static content found within the template's "repo/" folder is
  133  copied unchanged into the output folder for every repository.
  134  
  135  ### Scopes
  136  
  137  The variables available within each scope are listed here for reference:
  138  
  139  | Scope  | Variable        | Description                                           |
  140  | ------ | --------------- | ----------------------------------------------------- |
  141  | Index  | host            | The string from the `host` config option.             |
  142  |        | repositories    | A list of all of the git repositories.                |
  143  | Repo   |                 | *In addition to the variables from the index scope...*|
  144  |        | name            | The repository name, taken from its folder name.      |
  145  |        | description     | The repository's description (see below).             |
  146  |        | commits         | A list of the repository's commits.                   |
  147  |        | tree            | A list of the top-level folder's contents.            |
  148  |        | tree\_recursive | A list of *all* of the repository's contents.         |
  149  |        | tags            | A list of the refs corresponding to tags.             |
  150  |        | branches        | A list of the refs corresponding to branches.         |
  151  |        | readme          | The repository's readme file, if it has one.          |
  152  |        | license         | The repository's license file, if it has one.         |
  153  | File   |                 | *In addition to the variables from the Repo scope...* |
  154  |        | file            | A single file.                                        |
  155  | Commit |                 | *In addition to the variables from the Repo scope...* |
  156  |        | commit          | A single commit.                                      |
  157  
  158  As in [Jinja](https://jinja.palletsprojects.com), a list can be accessed with
  159  indexing, and attributes can be accessed using a dot notation. For example, a
  160  `repository` exposes an attribute called "name", so to access the name of the
  161  first repository from within the index scope you would do
  162  `repositories[0].name`.
  163  
  164  Here is the reference of attributes available on the variables that have them:
  165  
  166  | Object     | Attribute        | Description                                              |
  167  | ---------- | ---------------- | -------------------------------------------------------- |
  168  | repository | name             | The repository's name, taken from the folder name.       |
  169  |            | description      | The repository's description (see below).                |
  170  |            | head             | The current git commit.                                  |
  171  |            | updated          | The time when the current commit was committed.          |
  172  | file       | path             | The path the file relative to the repository root.       |
  173  |            | name             | The name of the file.                                    |
  174  |            | href             | The name of the HTML file for this file.                 |
  175  |            | contents         | The file's contents.                                     |
  176  |            | mode             | Directory, Plain, Executable, Symlink or Submodule.      |
  177  |            | mode\_octal      | Mode in octal form e.g. "00644" for plain files.         |
  178  |            | mode\_symbolic   | Mode in symbolic form e.g. ""-rw-r--r--" for plain files.|
  179  |            | is\_directory    | A boolean, useful for ginger conditionals.               |
  180  |            | is\_binary       | A boolean, tells you if the contents can be rendered.    |
  181  |            | tree             | A list of a directory's direct contents.                 |
  182  |            | tree\_recursive  | A list of *all* of a directory's contents.               |
  183  | ref        | name             | The tag or branch name.                                  |
  184  |            | commit           | The commit pointed to by the tag or branch.              |
  185  | commit     | id               | The SHA of the given commit.                             |
  186  |            | href             | The name of the HTML file for this commit.               |
  187  |            | title            | The commit message title.                                |
  188  |            | body             | The commit message body.                                 |
  189  |            | message          | The entire message, including both title and body.       |
  190  |            | diff             | The list of diff objects for this commit.                |
  191  |            | author           | The commit author.                                       |
  192  |            | committer        | The committer.                                           |
  193  |            | author\_email    | The email address of the author.                         |
  194  |            | committer\_email | The email address of the committer.                      |
  195  |            | authored         | The timestamp from when it was written.                  |
  196  |            | committed        | The timestamp from when it was committed to this branch. |
  197  |            | encoding         | The commit encoding.                                     |
  198  |            | parent           | The SHA of the parent commit.                            |
  199  | diff       | new\_file        | The name of the file after the diff.                     |
  200  |            | old\_file        | The name of the file before the diff.                    |
  201  |            | status           | The type of the change (see below).                      |
  202  |            | hunks            | The list of modified hunks for this file.                |
  203  | hunk       | header           | The hunk's header e.g. "@@ -2,44 +2,22 @@".              |
  204  |            | lines            | The list of lines  that form this hunk.                  |
  205  | line       | text             | The line, possibly prefixed by `+` or `-`.               |
  206  |            | class            | `"add"`, `"sub"` or `"def"` -- useful for CSS.           |
  207  
  208  Note:
  209  
  210  - Some attributes point to other objects that have attributes. For example,
  211    `branches[0].commit.parent` will work as expected.
  212  - "file" includes directories and symbolic links.
  213  
  214  ### Diffs
  215  
  216  Each commit has a corresponding list of diffs. Each diff corresponds to the
  217  changes made to a single file. A diff's `new_file` and `old_file` will differ
  218  only if the file was renamed. The `status` can be one of: Unmodified, Added,
  219  Deleted, Modified, Renamed, Copied, Ignored, Untracked, TypeChange.
  220  
  221  A diff contains a series of 'hunks' that represent contiguous blocks of text
  222  within the file that were modified in some way. Each has a header that
  223  indicates where the hunk is in the file. The hunks each also expose a list of
  224  _lines_, each of which exposes the text content of that line, as well as a
  225  'class'. The class is a convenience attribute that can be used to set the CSS
  226  class of the line without needing to parse the line text from within a template
  227  file. This makes it easier to style additions and subtractions from other (def
  228  for default) lines.
  229  
  230  ### Descriptions
  231  
  232  Each repository can have a description that is available in the templates. This
  233  is how the description for a given respository is determined:
  234  
  235  1. Look for a file at `repo/description`, and if found read from there.
  236  2. Otherwise, look for a file at `repo/.git/description`, and if found read from there.
  237  2. Otherwise, simply use the repository folder's name.
