    1  Documentation
    2  =============
    3  
    4  `gitserve` is a tool that reads from locally hosted or cloned git repositories
    5  and generates static HTML pages using their data.
    6  
    7  Usage
    8  -----
    9  
   10  ### Configuration file
   11  
   12  gitserve is configured using a [dhall](https://dhall-lang.org) config file. The
   13  [example config](https://github.com/m-col/gitserve/blob/master/config.dhall) is
   14  a good place to start, or alternatively create one in the current folder by
   15  running `gitserve -t`.
   16  
   17  It requires the following settings:
   18  
   19  | Setting              | Description                                         |
   20  | -------------------- | --------------------------------------------------- |
   21  | `repoPaths`          | A list of folders containing your git repositories. |
   22  | `templateDirectory`  | The folder containing the template (see below).     |
   23  | `outputDirectory`    | Where to put the generated files.                   |
   24  | `host`               | The host URL, which is needed for creating links.   |
   25  
   26  Then, pass the config file to gitserve.
   27  
   28  ### CLI
   29  
   30  See the CLI help message for usage:
   31  
   32      Usage: gitserve [-c|--config CONFIG] [-q|--quiet] [-f|--force] [-t|--template] 
   33                      [-v|--version]
   34        üêô Templated web page generator for your git repositories
   35  
   36      Available options:
   37        -c,--config CONFIG       Configuration file to use (Default: ./config.dhall).
   38        -q,--quiet               Suppress non-error output.
   39        -f,--force               Force regeneration of all files.
   40        -t,--template            Create a template and config in the current folder.
   41        -v,--version             Print the gitserve's version.
   42        -h,--help                Show this help text
   43  
   44  Note the "force" flag. By default, gitserve will not generate new output for
   45  commits to save time. This flag will force regeneration of all files, which
   46  would be needed if changes have been made to the template.
   47  
   48  Templates
   49  ---------
   50  
   51  A base template can be created in the current folder by running `gitserve -t`.
   52  This can be used as a starting point to creating your own template.
   53  
   54  ### Folder structure
   55  
   56  The files that gitserve outputs are generated using
   57  [ginger](https://ginger.tobiasdammers.nl) template files, which use the
   58  [Jinja](https://jinja.palletsprojects.com) templating language. If these are
   59  new to you, it may be enough to skim through some of the examples in the
   60  [templates](https://github.com/m-col/gitserve/tree/master/templates) folder,
   61  otherwise the ginger docs can be very helpful to see what is supported.
   62  
   63  Templates are a folder containing a number of ginger template files.  There are
   64  4 "scopes", each making available a unique set of variables storing information
   65  about the git repositories. Each template file has access to a single one of
   66  these scopes. The structure of the template folder determines the scopes of the
   67  files contained therein.
   68  
   69  To illustrate, this is the expected structure:
   70  
   71      templateDirectory/
   72          i_can_have_any_name.html
   73          and_there_can_be_any_number.html
   74          some_might_be_ginger_includes.html.include
   75          non_html_is_fine.css
   76          repo/
   77              inside_this_folder.html
   78              two_names_are_special.html
   79              file.html
   80              commit.html
   81  
   82  The top-level folder, here `templateDirectory`, is that which is specified in
   83  the config file.
   84  
   85  Files ending in ".html" directly within that folder have access to the *index
   86  scope*, and are each parsed exactly once and output into the output directory
   87  with the same name.
   88  "[include](https://ginger.tobiasdammers.nl/guide/syntax/statements/#include)"
   89  files are never copied but can be used to assist in templating. The remaining
   90  files, and any folders not called "repo", are copied as-is as static content,
   91  such as CSS, images, etc.
   92  
   93  The special folder "repo" has access to the *repo scope*, which exposes
   94  information pertaining to a single git repository. The template files contained
   95  within this folder are parsed and output once per git repository.
   96  
   97  The exceptions to this are the two special template files with the names
   98  "file.html" and "commit.html". These have access to the *file scope* and
   99  *commit scope* respectively, and are parsed and output once per file or commit.
  100  
  101  The resulting folder structure found in the `outputDirectory` will look like
  102  this (if the only specified git repository is gitserve):
  103  
  104      outputDirectory/
  105          i_can_have_any_name.html
  106          and_there_can_be_any_number.html
  107          non_html_is_fine.css
  108          gitserve/
  109              inside_this_folder.html
  110              two_names_are_special.html
  111              file/
  112                  LICENSE.html
  113                  Makefile.html
  114                  ...
  115              commit/
  116                  0a18f38bb5c398bd192a6268281fc6abefaedd63.html
  117                  0a7601059956d9c4d395f5d08e8cf48a515d080f.html
  118                  ...
  119          ...
  120  
  121  Note: symbolic links are considered static and will be copied unchanged and
  122  unresolved from template to output. This means that, for example, a symlink at
  123  `templateDirectory/index.html` pointing to `gitserve/index.html` will produce a
  124  symlink at `outputDirectory/index.html` with the same behaviour, with the
  125  effect that the served root index page will actually be the index page for the
  126  `gitserve` repository (if present).
  127  
  128  ### Scopes
  129  
  130  The variables available within each scope are listed here for reference:
  131  
  132  | Scope  | Variable     | Description                                           |
  133  | ------ | ------------ | ----------------------------------------------------- |
  134  | Index  | host         | The string from the `host` config option.             |
  135  |        | repositories | A list of all of the git repositories.                |
  136  | Repo   |              | *In addition to the variables from the index scope...*|
  137  |        | name         | The repository name, taken from its folder name.      |
  138  |        | description  | A description taken from a file called "description" in that folder.|
  139  |        | commits      | A list of the repository's commits.                   |
  140  |        | tree         | A list of the repository's files.                     |
  141  |        | tags         | A list of the refs corresponding to tags.             |
  142  |        | branches     | A list of the refs corresponding to branches.         |
  143  |        | readme       | The repository's readme file, if it has one.          |
  144  |        | license      | The repository's license file, if it has one.         |
  145  | File   |              | *In addition to the variables from the Repo scope...* |
  146  |        | file         | A single file.                                        |
  147  | Commit |              | *In addition to the variables from the Repo scope...* |
  148  |        | commit       | A single commit.                                      |
  149  
  150  As in [Jinja](https://jinja.palletsprojects.com), a list can be accessed with
  151  indexing, and attributes can be accessed using a dot notation. For example, a
  152  `repository` exposes an attribute called "name", so to access the name of the
  153  first repository from within the index scope you would do
  154  `repositories[0].name`.
  155  
  156  Here is the reference of attributes available on the variables that have them:
  157  
  158  | Object     | Attribute        | Description                                              |
  159  | ---------- | ---------------- | -------------------------------------------------------- |
  160  | repository | name             | The repository's name, taken from the folder name.       |
  161  |            | description      | A description taken from a file called "description" in that folder.|
  162  |            | head             | The current git commit.                                  |
  163  |            | updated          | The time when the current commit was committed.          |
  164  | commit     | id               | The SHA of the given commit.                             |
  165  |            | title            | The commit message title.                                |
  166  |            | body             | The commit message body.                                 |
  167  |            | message          | The entire message, including both title and body.       |
  168  |            | author           | The commit author.                                       |
  169  |            | committer        | The committer.                                           |
  170  |            | author\_email    | The email address of the author.                         |
  171  |            | committer\_email | The email address of the committer.                      |
  172  |            | authored         | The timestamp from when it was written.                  |
  173  |            | committed        | The timestamp from when it was committed to this branch. |
  174  |            | encoding         | The commit encoding.                                     |
  175  |            | parent           | The SHA of the parent commit.                            |
  176  | file       | path             | The path the file relative to the repository root.       |
  177  |            | href             | The HTML output path relative to `outputDirectory`.      |
  178  |            | contents         | The file's contents.                                     |
  179  |            | mode             | Directory, Plain, Executable, Symlink or Submodule.      |
  180  |            | mode\_octal      | Mode in octal form e.g. "00644" for plain files.         |
  181  |            | mode\_symbolic   | Mode in symbolic form e.g. ""-rw-r--r--" for plain files.|
  182  |            | is\_directory    | A boolean, useful for ginger conditionals.               |
  183  | ref        | name             | The tag or branch name.                                  |
  184  |            | commit           | The commit pointed to by the tag or branch.              |
  185  
  186  Note that some attributes point to other objects that have attributes.  For
  187  example, `branches[0].commit.parent` will work as expected.
